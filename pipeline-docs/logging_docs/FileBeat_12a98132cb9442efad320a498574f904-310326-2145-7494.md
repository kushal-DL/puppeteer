## **FileBeat** 

FileBeat is a lightweight shipper for forwarding and centralizing log data. Installed as an agent on your servers, FileBeat monitors the log files or locations that you specify, collects log events, and forwards them either to Elasticsearch or Logstash for indexing. 


![](logging_docs/markdown_converted/images/FileBeat_12a98132cb9442efad320a498574f904-310326-2145-7494.pdf-0001-02.png)


**FIPS** Linux requires package `filebeat-fips` , which is FIPScompliant. Windows package is not FIPS-compliant, but it has permanent exception. 

Here’s how FileBeat works: When you start FileBeat, it starts one or more inputs that look in the locations you’ve specified for log data. For each log that FileBeat locates, FileBeat starts a harvester. Each harvester reads a single log for new content and sends the new log data to libbeat, which aggregates the events and sends the aggregated data to the output that you’ve configured for FileBeat.

## 3.5. Multiline json messages 

In case your program logs in a pretty format and not a single line json, you can overcome the default behavior. 

```
filebeat:
  inputs:
  - type: log
    paths:
      - /tmp/sample.txt
    multiline.pattern: '^{'
        multiline.negate: true
        multiline.match: after
```

## 3.2. Load balance filebeat output 

Why to balance the load on client? In all instances of our Logging Service, we use F5 pool or AWS ELB with least connection algorithms. This equally spread connections across our stack. However, some connections send less than 1 message per second, others send 1000 or even more messages per second. This way, our logstashes doesn't work in optimal mode. Then it may (and did many times) happen, your filebeats are not accepted to send data, as logstash is overloaded. If you use loadbalancer on filebeat, it will automatically open 2 (or more) connections = will connect to at least 2 logstashes and therefore send half load to each. 

So, if your application generates huge amount of logs (ie. if from single host, you send more than 2k-5k logs/min ~ 50/s), we recommend to do local load balancing. This is a trick suggested by Elastic support team - repeat the same VIP in **output.logstash.hosts** property several times (and add loadbalance: true property). This way, your application won't overload one single logstash, but the load will be spread more across the stack. Also, this will help your application to reduce latency while Logging Service is more loaded. 

Example loadbalance config file snippet for Filebeat (for Logging Service in ATM): 

```
#/etc/filebeat/filebeat.yml
filebeat:
  prospectors:
    -
      paths:
        - /var/log/app/abc.log*
      input_type: log
      include_lines: ['^{']
output:
  logstash:
    hosts: ["lp-index.service.cnqr.tech:10300", "lp-
index.service.cnqr.tech:10300"]
    ttl: 15m
    loadbalance: true
    ssl:
       enabled: true
       verification_mode: none
```

**Advanced Note** : If your filebeat send more than 500 messages per second you can/should add even more VIPs in the hosts array. Please, always check with the logging team for more information and guidance ( #ask-logging )
