## **Access from Concur Global** 

Concur Global REST API links (means concur.global domain - like htps://lp-search-rest.integration.concur.global) DO NOT SUPPORT vault certificate authentication by design. We are NOT able provide such access to you either. 

In case you need REST API access to Logging Service from Concur Global network (SAP office), you can use short term API keys instead. Elasticsearch API keys are generated self-service, through kibana (Menu  Stack Management  Security/API Keys). 

**But please, be aware that ALL keys are invalidated by our internal process after 24 hours of their creation!** These belong to the user who created them, and we will NOT provide any for long term or for service users. In such case, use Vault certificates described below, where the process runs from inside the environment. 

## **Server Certificates** 

We don't require REST API clients to validate server certificates. On the Logging services REST API endpoints side we use server certificates issued by Concur DA Internal Certificate Automation API or Vault Enterprise. 

If you face a server certificate error, get the corresponding Root CA certificate using hints at Internal Certificate Automation API or https://pages.github. concur.com/disco/docs/enterprise/#anamepkirootapkirootcertificates, include it in your keychain, and make it trusted. 

## **Client verification** 

Users require their client certificate issued by the environment's Vault and use it for authentication. The requests require the whole client CA chain attached in that order: 

1.  client certificate 2.  Vault intermediate certificate(s) - especially including the client Vault Namespace intermediate CA cert! 3.  the Vault Root CA certificate 

Some clients like `PostMan` or older `curl` send only the first of these certificates so the REST server cannot reconstruct the full CA chain to the Vault root to verify the client cert, because Vault team introduced hundreds of intermediate CAs per each Vault NameSpace we have no way to sync with. 

The **bash** snippet: 

```
make_csr () {
  declare subject="$1"
  openssl req -new -nodes \
    -newkey rsa:"${BITS}" \
    -keyout client.key \
    -out    client.csr \
    -subj   "${subject}"
  openssl req -noout -text -in client.csr
}
get_token () {
  no_proxy="*" curl -skL \
    -d "$(jq -nc --arg password "${VAULT_PASS}" '{password: $password}')" \
    "${VAULT_ADDR}/v1/auth/ldap/login/${VAULT_USER}" \
  | jq -r '.auth.client_token' > vault.token
  echo "VAULT_TOKEN = $(<vault.token)" >&2
}
get_response () {
  declare path="$1"
  declare response="$(
    no_proxy="*" curl -vkL \
      -d "$(jq -csR '{csr:.}' < client.csr)" \
      -H "X-Vault-Token: $(<vault.token)" \
      "${VAULT_ADDR}${path}" > response.json
  )"
  jq -S . <response.json 2>&1
}
get_crts () {
  jq -r .data.certificate <response.json > client.crt
  openssl x509 -text -noout -in "client.crt"
  jq -r '.data.ca_chain[]' <response.json > chain.crt
  cat client.crt chain.crt > all.crt
}
# settings
VAULT_ADDR=https://vault.service.cnqr.tech
VAULT_USER=...
VAULT_PASS=...
VAULT_NAMESPACE=...
VAULT_ROLE=...
BITS=2048
CLIENT_FQDN=....
# validate requirements
for binary in openssl, curl, jq; do
  which "${binary}" >/dev/null \
    || echo "Error: Missing the dependency: ${binary}">&2
done
# create client.key and client.csr
make_csr "/CN=${CLIENT_FQDN}"
# login to Vault, create vault.token
get_token
# issue the certificate, create response.json
get_response "/v1/${VAULT_NAMESPACE}/pki/client/sign/${VAULT_ROLE}"
# from the response create client.crt, chain.crt and all.crt
get_crts
```

## **REST API with** **`curl`** 

Basic and advanced searches/aggregations can be run using the REST API with `curl` : 

(note that **`'*:'`** prefix is needed only in the SEA at this point) 

```
curl \
  --cacert ./chain.crt \
  --key    ./client.key \
  --cert   ./client.crt \   https://lp-search-rest.us2.concur.global/*:log-2/_search -d'
        {
          "size":10,
          "query": {
            "query_string": {
              "query": "application:EMT"
            }
          }
        }
        '
```

## **REST API with Elasticsearch Java REST client library** 

Elasticsearch provides two java REST client one high level and one low level, to instantiate a high-level one you will need to prepare a low-level client first. 

Full documentation available at https://www.elastic.co/guide/en/elasticsearch/client/java-rest/7.17/java-rest-overview.html 

```
public class maintest {public static void main(String[] args)throws IOException,KeyStoreException,
NoSuchAlgorithmException,KeyManagementException,java.security.cert.CertificateException,java.security.
UnrecoverableEntryException {
        final SSLContext sslContext = SSLContexts.custom()
                .loadTrustMaterial(null, new TrustStrategy(){
                    public boolean isTrusted(X509Certificate[] chain, String authType)
                            throws CertificateException {
                        return true;
                    }
                })
                .build();
        RestClientBuilder builder = RestClient.builder(new HttpHost("lp-search-rest.us2.concur.global", 443,
"https"))
                .setHttpClientConfigCallback(new RestClientBuilder.HttpClientConfigCallback() {
                    public HttpAsyncClientBuilder customizeHttpClient(HttpAsyncClientBuilder httpClientBuilder)
{
                        return httpClientBuilder.setSSLContext(sslContext);
                    }
                });
        RestClient lowLevelRestClient = builder.build();
                Response response = lowLevelRestClient.performRequest("POST", "/*:log-2/_search", Collections.
singletonMap("pretty", "true"));
        System.out.println(EntityUtils.toString(response.getEntity()));
    }
```

To instantiate a High-level REST client at this point is just: 

```
RestHighLevelClient client =
        new RestHighLevelClient(lowLevelRestClient);
```

This gives you access to many complex builders for operations, described in Elasticsearch Java REST client documentation. 

## **REST API with Elasticsearch Python REST client library** 

The Elasticsearch Python library gives you all the information necessary to create a new Elasticsearch client instance and run searches. 

To create a new REST client in python you will need the  Concur Cloud Services or HashiCorp Vault Root CA certificate from Internal Certificate Automation API - Deprecated#RootandIntermediateCACertificates or HashiCorp Vault.

## **Scroll API** 

Using the Java high-level client you get access to the scroll API, this API is useful to fetch a batch of documents from Elasticsearch. 

The scroll execution stops when the response of the last request is empty. 

Is a good practice to clear the scroll request at the end of the search or in case of errors, this allows the cluster to free resources for future requests. 

```
RestHighLevelClient client =
        new RestHighLevelClient(lowLevelRestClient);
final Scroll scroll = new Scroll(TimeValue.timeValueMinutes(1L));
SearchRequest searchRequest = new SearchRequest("*:log-2");
searchRequest.scroll(scroll);
SearchSourceBuilder searchSourceBuilder = new SearchSourceBuilder();
searchSourceBuilder.query(matchQuery("application", "EMT")).size(100);
searchRequest.source(searchSourceBuilder);
SearchResponse searchResponse = client.search(searchRequest);
String scrollId = searchResponse.getScrollId();
SearchHit[] searchHits = searchResponse.getHits().getHits();
for (SearchHit hit : searchHits) {
    System.out.println(hit.getSource().toString());
}
```

```
while (searchHits != null && searchHits.length > 0) {
    SearchScrollRequest scrollRequest = new SearchScrollRequest(scrollId);
    scrollRequest.scroll(scroll);
    searchResponse = client.searchScroll(scrollRequest);
    scrollId = searchResponse.getScrollId();
    searchHits = searchResponse.getHits().getHits();
    System.out.println(scrollId);
    for (SearchHit hit : searchHits) {
        System.out.println(hit.getSource().toString());
    }
}
```

```
ClearScrollRequest clearScrollRequest = new ClearScrollRequest();
clearScrollRequest.addScrollId(scrollId);
ClearScrollResponse clearScrollResponse = client.clearScroll(clearScrollRequest);
```

More examples and features are described on the Search Scroll API.

## **Query without time filter** 

You can use Kibana search query in REST API calls also using query_string. 

NOTE: if you use quotes in your query, they have to be escaped here... 

```
POST /{index}/_search
{
  "query": {
    "query_string": {
      "query": "roletype: xyz AND application:foo"
    }
  }
}
```

## **Query with time filter** 

This query is finally the real example of how to correctly write queries: 

```
POST /{index}/_search
{
  "query": {
    "bool": {
      "must": [
        {
          "query_string": {
            "query": "roletype: xyz AND application: foo"
          }
        },
        {
          "range": {
            "@timestamp": {
              "gte": "now-15m",
              "lte": "now"
            }
          }
        }
      ]
    }
  }
}
```

Few notes about this query: 

please notice that compared to previous query, we put query_string under bool and must. This is the way how to combine queries and range and filters. In this case all (query_string and range) criteria must be met. To negate, you may include must_not array under bool.
