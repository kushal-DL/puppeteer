## Introduction 

The aim of the AWS Lamba Logging is to have logging from lambdas as easy as possible for the end user. User should not install filebeat into lambda to be able to send the lambda logs successfully. 

Logging from Lambda has the same requirements as any other applications have. That means: 

We do accept ONLY single-line JSON formatted log Simply put it to stdout 

For all types of logs (applications logs, audit, appsec), you have to meet all requirements (like required fields, ...). Described here: How to format log data 

If all requirements are met, your data will be collected and processed - **in ATM environments** ! 

## How it works 

There is a pipeline AWS Central Logging that collects all raw lambda logs and writes it into an S3 bucket. The pipeline is supported by the CLZ team and questions about it should be asked in #ask-clz. 

Logging Service logstashes then 'listen' to any changes in this bucket by SNS/SQS and collect all logs that are json for further validation and processing. 

## Shared responsibility model 

AWS Lambda Logging ingestion pipeline uses shared responsibility model: 

CLZ team and their solution is responsible to collect all STDOUT logs and deliver them to central S3 bucket (ie. `concur-devtest-centrallogs` in integration), directory `/lambda` . 

Logging team is responsible to collect all valid JSON documents from this directory, and process it 

In case of investigating missing data, please consider above information
