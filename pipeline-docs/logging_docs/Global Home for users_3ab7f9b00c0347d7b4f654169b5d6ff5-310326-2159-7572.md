## What is going on 

We are going to migrate away from concursolution.com domain, to only supported concur.global domain. This migration is part of our effort to streamline and unify our platform’s identity across all environments, ensuring consistency, compliance and improved service reliability. 

The transition will be seamless for all users, as the authentication is not changing. 

## **What to do with bookmarks and short links** 

Users need to start using the new URL instead of the old one. That requires to change all bookmarks, as the old url will stop working few days after we fully migrate to new domain. 

Short links will normally work with the new URL. 

Example: (this is not real short URL - just an example to explain) 

If your old short url looks like this: 

https://lp-search.us.concursolutions.com/app/a/b/cdefg 

Then update the domain, and keep the whole endpoint as it was, like here: https://lp-search.us2.concur.global/app/a/b/cdefg 

## **Backword incompatibility** 

REST API on the new domain will NOT support vault certificate authentication. There are usually two use-cases: 

accessing REST API from inside the datacenter - using https://lp-search-rest.service.cnqr.tech URL and Vault certificate. This will continue to work without any change 

accessing REST API from developer laptop - using https://lp-search-rest.{env}.concursolutions.com. This one will STOP working after the migration to the concur.global domain. The reason is, Concur Global doesn't support direct authentication and specifically long term keys /certificates. 

In this case, users will be allowed to generate (directly in Kibana UI) short-living API key, which will last max 24hours, and will allow them to develop solution on their dev laptop.

## FAQ 

## **Why it's changing?** 

Onboard to the modern, well supported network access to production environments using Global Home. Simplify deployment, so make the endpoints best available, stable, cheaper, safer. Improve security by hiding public endpoints behind a VPN. 

## **Do I need to change anything on the data ingestion and processing side?** 

No, this change is only on data access side - Kibana UI, and REST API (outside the DC) 

## **Do I need to change something** 

Yes, update all your bookmarks and links to kibana instances. Automated watches will be transitioned for you automatically. 

## **Why I see two different OKTA authentication links in kibana now?** 

Migrating the service to new domain requires full cluster restart. That is time-intensive operation, and since we want to minimize time when users are impacted, we are preparing our clusters in advanced. That includes two authentication methods available, but only one is actually functional. 

## **When new endpoints are available?** 

They are already available, but OKTA authentication forwards you to the old endpoint 

## **When old endpoints will be decommissioned?** 

On November 20th, 2025.
