## **Index patterns - which one to use** 

**data alias** 

We support three main data types within data_version 2: log, metric and txn. There is an index pattern for each of them (*:log-2, *:metric-2 and *:txn-2). Most of users however use another one: *:data-2. Data index pattern is only a virtual pattern which points to all log and all metric indexes. This was added long time ago, to support easier search for users. But now, it cause not necessary load to the system. If you know where your data are stored, use that one. If you are not sure, try it with one, and you'll see which one shows data on your query. 

## **Timelion usage** 

Timelion is really great tool and you can do quite crazy data manipulation there. But if you do it wrong, this is biggest 'killer' of service performance. For reference: https://www.elastic.co/guide/en/kibana/7.17/timelion.html (v5: https://www.elastic.co/guide/en/kibana/5.6/timelion-create.html) 

Timelion syntax starts with .es() function. No parameter is required. If you put just a string inside, it is meant to be query. If you don't put anything, the query is by default everything '*'. BUT, if you don't specify which index you want to search over, ALL indexes are searched. And this is most common mistake - users just put query and it works!! 

Example: 

```
.es('application: abc AND host: *pr*')                                                 // heavy query with
wildcards, applied to ALL indexes (ie. in US_comm ~910 indexes)
```

```
.es(index=*:metric-2, q='application: abc AND host: *pr*')                // still heavy query due to
wildcards, but applied to more focused content (ie. in US_comm ~58 indexes)
```

## **Wildcards usage** 

It is common practice to use wildcards when searching data. And that is valid use-case. But only when you search for data. It is definitely bad practice to use in visualizations. This simply refer to a problem: you either don't know how to use it properly, or data are not well formatted. 

All string type fields use two different methods for storing data: keyword or text: 

## **Keyword fields** 

Anything you put inside keyword type field, is taken as one single value - token. All characters as they are. If you need to search only for part of it, you need to use wildcard. 

Aggregations can be computed ONLY on keyword fields! 

Example: "keyword_field": "/my/great/endpoint". For searching all documents where keyword_field field start with '/my', I can use query: keyword_field: 

/my*. 

## **Text (analyzed) fields** 

Value used in text field is analyzed, words (tokens) are taken and indexed. Means, you don't search over whole value, but tokens inside. We always use standard analyzer: https://www.elastic.co/guide/en/elasticsearch/reference/5.6/analysis-standard-analyzer.html 

Example:  "text_field": "/my/great/endpoint" For searching all documents where text_field field start with '/my', I can use query: keyword_field: /my. As you can see - we don't need to use wildcard. Wildcard actually can return completely different results, as it will search over ALL tokens in ALL documents with this field, where some token start with my*. You can expect false results. 

Fields and their mapping can be found here: Application Log Format (v2), where not_analyzed = keyword, 

analyzed (80 tokens) means text field, but only first 80 tokens (words) are indexed, .analyzed means the field itself is (usually) of type keyword, but has inner field which is analyzed 

## **.analyzed** 

As mentioned above, some fields (only core - all custom buckets are by default only keywords) has inner .analyzed field. Ie. cte.category. In that case, you can write queries against cte.category as well as cte.category.analyzed. Results will be different - as described just above. Use appropriately. 

## **Typical mistakes/weirdnesses** 

your visualization use *:data-2 alias, but first part of your query is: **_type: log-2** , or **level: error** . In both these cases, only data from *:log-2 are valid. Solution: change index pattern to *:log-2
