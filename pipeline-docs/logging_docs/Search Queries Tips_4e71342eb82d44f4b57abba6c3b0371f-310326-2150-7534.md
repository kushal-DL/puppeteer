## **Search Queries Tips** 

This page gives hints and tips how to write Kibana queries , in accordance to current data mapping in use in Logging Service. 

## **Mapping** 

Current mapping, applied to all Logging Service instances, has following specifics: 

all fields are lower-cased during indexing time 

all string fields are by default not analyzed (actually, lowercase analyzer is applied, but the value is not split to terms (tokens)) 

- some string fields contain '.analyzed' nested field, i.e. these: application, category, subcategory, method, service, action (resp. all these in cte bucket) 

we usually support two data mapping versions at the same time, current "active" one, and one version behind, so you might have different queries between different mapping versions. 

## **Possible problems and solutions** 

term filters are case sensitive. If you want to properly use these filters, lowercase filter value if you dont see your field indexed, try to refresh fields list in Kibana, see Kibana #ReloadFieldListWhenYouAddNewFieldstoLogstash , if this does not help, then field is really not indexed and we need to update the mapping, so please create LOG JIRA ticket for us to do so. 

## **Index patterns differencies in SEAPR1** 

Because of service architecture differences between SEA and all other jurisdictions, there is different way in SEA how to use index-patterns in Kibana /REST APIs. 

In SEA, we store data in multiple clusters (main, coresrv). And lp-search-sea.concur.com endpoint points to virtual cluster, which only search on remote clusters. Normal index pattern convention is: 

```
{index_prefix}-*
```

In SEA, we have to add even cluster_name with this format: 

```
{cluster_name}:{index_prefix}-*
```

where cluster_name is either 'main' or 'coresrv'. Or all remote clusters, when using star. 

These are example of possible index-patter usage in SEA: 

```
*:data-*
main:data-*
coresrv:log-*
*:metric-*
```

## **Querying and filtering tips** 

See query string syntax docs at https://www.elastic.co/guide/en/elasticsearch/reference/current//query-dsl-query-string-query.html#query-string-syntax . Its based on Lucene query syntax, see https://lucene.apache.org/core/2_9_4/queryparsersyntax.html . 

Few samples: 

How to ask for particular field values 

```
application:counter
```

if you need to use a wildcard to properly find all your data, you can use it like: 

```
application:queue*
```

Some characters needs to be escaped for proper search (mainly space and colon, if they should be part of searched value) 

```
exception:Exception\:\ Error\ caught*
```

Some data providers are using new data structure, and others old. To be sure you get all results, combine the query: 

```
category:value OR cte.category:value
```

If this is part of longer query, put the subquery in brackets: 

```
application:apps AND (category:value OR cte.category:value) AND ...
```

If you use filter (terms or field) with category or some other field that changed, create a new queryfilter (by clicking on the + sign at the end of filters) with above query. Then delete (or deactivate) old filter.
